浏览器缓存有两种机制：过期机制和验证机制

浏览器缓存的实现方式主要有两种：HTTP 和 ServiceWorker

# 1. 浏览器的缓存机制

对于浏览器来说，这些规则是在 HTTP 协议头部和 HTML 页面的 Meta 标签中定义的，他们分别是从新鲜度和校验两个维度来规定浏览器是否可以直接使用缓存中的副本，还是需要去源服务器中获取新版本

过期机制：指的是缓存副本的有效期，一个缓存的副本是足够新的，是有效的需满足两个条件

  • 含有完整的过期时间控制头信息（HTTP 协议报头），并且仍在有效期内
  • 浏览器已经使用过这个缓存副本，并且会在一个会话中已经检查过新鲜度（即服务器上的资源是否有发生改变），只要满足一种，浏览器就会直接从缓存中获取副本进行渲染

校验值（验证机制）：服务器返回资源的时候有时在控制头信息带上资源的实体标签（Etag），它可以用做浏览器再次请求过程中的校验标识，如果发现标识不匹配，说明资源已经被修改了或者过期了，浏览器需要重新获取资源内容

# 2. HTTP 缓存

使用 HTTP 缓存最大的问题往往不在于资源缓存在什么位置或者如何读写资源，而是说尽可能地让浏览器从缓存中获取资源，但又要保证被使用的缓存与服务器资源保持一致

# 2.1 强缓存：

强制缓存是在浏览器加载资源的时候，先直接从缓存中查找请求结果，如果不存在该结果，则直接向服务器发起请求

Expires HTTP1.0 中可以使用响应头部字段 Expires 来设置缓存时间，它对应一个未来的时间戳，客户端第一次请求时，服务器会在响应头部添加 Expires 字段，当浏览器再次发送请求时，先会对比当前时间和 Expires 对应的时间，如果比 Expires 早，就直接使用缓存，如果晚就发起请求

使用 Expires 会产生一个问题：那就是服务端和浏览器的时间很可能不同，因此这个缓存过期时间容易出现偏差，同样，客户端也可以通过修改系统的时间来继续使用缓存或提前让缓存失效

为了解决这个问题，HTTP1.1 提出了 Cache-Control 响应头部字段

Cache-Control 常用的几个值

  • no-cache 表示使用协商缓存，即每次使用缓存前必须向服务器确认缓存资源是否更新
  • no-store 禁止浏览器以及所有中间缓存存储相应内容
  • public 共有缓存，表示可以被代理服务器缓存，可以被多个用户共享
  • private 私有缓存，不能被代理服务器缓存，不可以被多个用户共享
  • max-age：以秒为单位的数值，表示缓存的有效时间
  • must-revalidate：当缓存过期时，需要去服务器校验缓存的有效性

注意：max-age 的优先级高于 Expires 的优先级，同时出现，浏览器会使用 max-age 为主

# 2.2 协商缓存

协商缓存的更新策略是不在指定缓存的有效时间了，而是浏览器直接发送请求到服务器进行确认缓存是否跟新，如果请求响应返回的 HTTP 状态为 304，则表示缓存仍然有效，控制缓存的难题就从浏览器转移到了服务端

Last-Modified 和 If-Modified-Since

服务器判断缓存有没有过期，只能将双方的资源进行对比，若浏览器直接把文件发给服务器进行对比的话，网络开销很大，而且还不如直接请求拿数据的好，所以有一种解决的办法就是通过响应头部 Last-Modified 和请求头部 If-Modified-Since 比对双方资源的修改时间

具体工作流程

  • 浏览器第一次请求资源，服务端在返回资源的响应头中加入 last-Modified 字段，该字段表示这个资源在服务端上最近的修改时间
  • 当浏览器再次向服务端请求该资源时，请求头部带上之前的服务端返回的修改数据，这个请求头叫 If-Modified-Since
  • 服务端再次收到请求，根据请求头 If-Modified-Since 的值，判断相关资源是否有变化，如果没有，则返回 304 Not Modified，而且不返回资源，浏览器使用资源缓存，否则就正常返回资源内容，且更新 Last-Modified 的值

存在的问题：

  • 精度问题：Last-Modified 的时间精度是 s，如果在一秒内发生了修改，他都会重新请求服务器获取数据
  • 准度问题：一个文件被修改，然后又被还原，这样的话也会重新请求服务器

ETag 和 If-None-Match

为了解决精度问题和准度问题，HTTP 提供了一种不依赖于修改时间，而依赖于文件哈希值的精确判断缓存的方式，那就是响应头部 ETag 和请求头部 If-None-Match

具体工作流程

  • 浏览器第一次请求资源时，服务端在返回响应头中加入了 ETag 字段，Etag 字段值为该资源的哈希值
  • 当浏览器再次跟服务端发这个请求时，在请求头上加上 If-None-Match，值为之前响应头部字段 ETag 的值
  • 服务端收到请求，将请求头的 If-None-Match 字段的值和响应资源的哈希值进行对比，如果两个值相同，那么说明没变化，就返回 304，否则返回资源内容，无论是否发生改变，都会将计算出的哈希值放入响应头的 ETag 中

存在的问题：

  • 计算成本：生成哈希值相对读取文件修改时间而言是一个开销较大的操作，对于大文件更甚
  • 计算误差：HTTP 没有规定哈希值的计算方法，所以不同服务端有自己的计算方式，可能会导致同一个资源在两台不同的服务端产生的 ETag 可能时不一样的
# 2.3 HTTP 缓存的注意点
  • 强制缓存的优先级高于协商缓存，协商缓存中的 ETag 优先级比 Last-Modified 高
  • 在精度上，Etag 优于 Last-Modified
  • 在性能上，Last-Modified 由于 ETag

# 3. Service Worker

Service Worker 是浏览器在后台独立于网页运行的脚本，也可以这么理解，他是浏览器和服务端之间的代理服务器，ServiceWorker 非常强大，可以实现包括推送通知和后台同步等功能，最主要的功能还是实现离线缓存

使用限制：

  • 在 ServiceWorker 中无法直接访问 DOM，但是可以通过 postMessage 接口发送的消息来与其控制的页面进行通信
  • ServiceWorker 只能在本地环境下或 HTTPS 网站中使用
  • ServiceWorker 有作用域的限制，一个 ServiceWorker 脚本只能作用于当前路径及其子路径
  • ServiceWorker 属于实验性功能，所以兼容方面会存在一些问题

使用方法：在使用 ServiceWorker 脚本之前先要通过注册的方法来加载它

浏览器获取到 ServiceWorker 脚本之后会进行解析，解析完会进行安装，可以通过监听”install“事件来监听安装，这个事件会在第一次加载脚本的时候触发，要让脚本能够监听浏览器的网络请求，还需要激活脚本

在脚本被激活后，我们就可以通过监听 fenth 事件来拦截请求并加载缓存的资源了，在捕获到 fenth 请求时，会返回 event.repondWith 函数的 Promise 对象的结果，在这个 Promise 对象中，我们可以通过 cache.match 来和当前请求对象进行匹配，如果匹配上了就返回匹配的结果，否则会返回请求结果并缓存

# 4. 缓存位置

当缓存命中或者协商缓存在服务器中返回 304 的时候，我们会直接从缓存中获取资源，而浏览器中的缓存位置一共有四种，按优先级从高到低分别是：

  • Service Worker
  • Memory Cache
  • Disk Cache
  • Push Cache

如果以上四种缓存都没有命中的话，那么只能发起请求来获取资源了

Service Worker

Service Worker 借助了 Web Worker 的思路，即让 js 运行在主线程之外，由于他脱离了浏览器的窗体，因此无法直接访问 DOM，虽然如此，但他们仍然能帮助我们完成很多有用的功能，比如离线缓存，消息推送和网络代理等功能，其中离线缓存就是 Service Worker Cache

Memory Cache 和 Disk Cache

Memory Cache 指的是内存缓存，从效率上讲他是最快的，但是从存活时间来讲时最短的，当渲染进程结束后，内存缓存也就不存在了，一旦我们关闭 Tab 页面，内存中的缓存也就被释放了

Disk Cache 值得时磁盘缓存，从存储效率上来说，比内存缓存要慢，但是他的优势在于存储容量和存储时长

  • 比较大的 JS，CSS 文件就会放进磁盘，反之就放进内存
  • 内存使用率比较高的时候，文件优先进入磁盘

Push Cache

即推送缓存，这是浏览器缓存的最后一道防线，他是 HTTP2 的内容，虽然现在应用并不是很广泛，但随着 HTTP2.0 的推广，它的应用越来越广泛

# 5. 总结

首先通过 Cache-Control 验证缓存时候可用

  • 如果强缓存可以使用，那就直接使用
  • 否则进入协商缓存，即发送 HTTP 请求，服务器通过请求头的 If-Modified-Since 或者 If-None-Match 字段检查资源时候更新
    • 没有更新，返回 304，告诉浏览器直接从缓存中获取资源
    • 更新了，返回资源和 200
