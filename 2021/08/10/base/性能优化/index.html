# 1. 浏览器的组成

# 1.1 浏览器内核

浏览器中有一个最重要的模块，它主要的作用是把一切请求回来的资源变成可视化的图像，这个模块就是浏览器内核，通常也称作渲染引擎

IE（Trident）

Safari（Webkit）：Webkit 本身主要由两个小引擎构成的，一个是渲染引擎 “WebCore”，另一个是 javascript 解释引擎 “JSCode”，他们都是从 KDE 的渲染引擎 KHTML 以及 javascript 解释引擎 KJS 衍生来的

Chrome（Blink）：在 13 年发布的 Chrome 28.0 版本开始，Chrome 放弃了 Chromium 引擎转而使用最新的 Blink，比上一代的引擎精简了代码，改善了 DOM 框架，也提高了安全性

Opera（Blink）：4-6 版本使用 Elektra 引擎，7.0 使用 Presto 渲染引擎，13 年 2 月防疫 Persto 转而使用 Chromium，又转而使用 Blink 引擎

Firefox（Gecko）

# 1.2 进程和线程

进程：程序的一次执行，它占有一片独立的空间，是操作系统执行的基本单元，一个进程至少有一个运行的线程：主线程（在进程启动后自动创建），一个进程中也可以同时运行多个线程，一般来说程序时多线程运行的，一个进程内的数据可以提供给多个线程共享，多个进程之间的数据是不能直接共享的

线程：是进程的一个执行单元，是 CPU 调度的最小单元，程序运行的基本单元线程池：保存多个线程对象的容器，实现线程对象的反复利用

JS 引擎是单线程运行的！

# 1.3 现代浏览器：多进程，多线程模型

过去：当你通过浏览器打开很多页面的时候，如果其中一个页面不响应了或者崩溃了，你打开的所有页面都会得不到响应，可能还有一些页面包含着一些没有保存的或者没有发送的信息

如何解决：采用多进程模型，它可以避免因单个页面不响应或者崩溃的时候对整个浏览器的影响，当第三方插件崩溃时，也不会影响整个浏览器的稳定性，而且还安全

浏览器有哪些进程：

  • Browser 进程：浏览器的主进程，负责浏览器界面的显示和各个页面的管理，负责其他进程的创建和销毁
  • Renderer 进程：网页渲染进程，负责网页的渲染，可以有多个
  • 各种插件进程
  • GPU 进程
# 1.4 每个进程内部又有很多的线程

多线程的目的主要是保持用户界面的高度响应

例如：为了不让 Broswer 进程的 UI 线程被其他耗时的操作（大文件，本地文件读写）所阻塞，我们会把这些操作放到分线程中去处理，在 Renderer 进程中，为了不让其他操作阻止渲染线程的高速执行，我们通常会将线程管线化

# 2. 浏览器的主要模块

# 2.1 浏览器的渲染引擎

一个渲染引擎主要包括：HTML 解析器，CSS 解析器，javascript 引擎，布局 layout 模块，绘图模块

  • HTML 解析器：解析 HTML 文档的解析器，主要是将 HTML 文本解析成 DOM 树
  • CSS 解析器：为 DOM 中的各个元素对象计算出样式信息，为布局提供基础设施
  • JS 引擎：使用 JS 代码可以修改网页的内容，也能修改 css 的信息，JS 引擎能解释 JS 代码
  • 布局模块：在 DOM 创建之后，Webkit 需要将其中的元素对象同样式信息结合起来，计算他们的大小位置等布局信息
  • 绘图模块：使用图形库将布局计算后的各个网页的节点绘制成图像结果

大致的渲染过程：

  • 遇见 HTML 标记，调用 HTML 解析器解析为对应的 token（一个 token 就是一个标签文本的序列化）并创建 DOM 树
  • 遇见 style/link 标记，调用相应的解析器处理 CSS 标记，并构建出 CSS 样式树
  • 遇见 script 标记，调用 JS 引擎，处理 script 标记，绑定事件，修改 DOM 树 / CSS 树等
  • 将 DOM 树和 CSSOM 树合并成一个渲染树
  • 根据渲染树来渲染，以计算每个节点的集合信息（这一个过程需要依赖 GPU）
  • 最终将各个节点绘制到屏幕上去
# 2.2 样式的渲染注意点

style 标签中的样式是由 html 解析器（异步）来进行解析的，容易出现闪屏现象（html 解析器先解析完了 html 结构，并且把结构渲染了，这是解析的 style 才被渲染，就会出现闪屏现象）

link 标签引入的样式是由 css 解析器去解析，并且是同步解析的，css 解析器会阻塞页面的渲染（理由这一点来避免闪屏问题）

# 3. 关于阻塞

# 3.1 CSS 阻塞

只有 link 引入的外部 css 才能够产生阻塞

  • style 标签中的样式：
    1. 由 html 解析器进行解析
    2. 不阻塞浏览器渲染（可能会产生 “闪屏现象”）
    3. 不阻塞 DOM 解析
  • link 引入的外部 css 样式（推荐）
    1. 由 css 解析器进行渲染
    2. 阻塞浏览器渲染（可以利用这个阻塞来避免 “闪屏”）
    3. 阻塞其后面的 js 语句的执行
    4. 不阻塞 DOM 的解析（绝大多浏览器的工作方式）
  • 优化核心理念：
    1. 使用 CDN 节点进行外部资源的加载速度
    2. 对 css 进行压缩（利用打包工具，webpack，gulp）
    3. 减少对 http 请求数，将多个 css 文件合并
    4. 优化样式表的代码
# 3.2 js 阻塞
  • 阻塞后续的 DOM 解析：浏览器并不知道脚本后续的内容，如果先去解析了下面的 DOM，而随后的 js 删除了所有的 DOM，那么浏览器就相当于做了无用功，所以索性停住，等脚本执行完了，浏览器再继续向下解析 DOM
  • 阻塞页面的渲染：js 也可以给 DOM 设置样式，浏览器等该脚本执行完毕，渲染出一个最终的结果
  • 阻塞后续的 js 的执行：维护依赖关系
# 3.3 阻塞备注
  • css 的解析和 js 的执行是互斥的，css 解析的时候 js 会停止执行，js 执行的时候 css 会停止解析
  • 无论 css 阻塞还是 js 阻塞，都不会阻塞浏览器请求发出去的工作模式，只要是涉及到网络请求的内容都会先发请求去获取资源，至于资源回来了什么时候用，由浏览器自己协调
  • Webkit 和 Firefox 都进行了预解析这项优化，在执行 js 脚本时，浏览器的其他线程会预解析文档的其余部分，找出并加载需要通过网络加载的其他资源，这样资源就可以并行连接上加载，从而提高总体的速度，预解析不会修改 DOM 树

# 4. CSS 图层

浏览器在渲染一个页面时，会将页面分为很多层，图层有大有小，每个图层有一个或者多个节点，在渲染 DOM 的时候，浏览器所作的工作实际上是：

  • 获取 DOM 后分割为多个图层
  • 对每个图层的节点计算样式结果（Recalculate style -- 样式重计算）
  • 为每个节点生成图形和位置（重排）
  • 将每个节点重新绘制填充到图层位图中（重绘）
  • 图层作为纹理上传到 GPU
  • 组合多个图层到页面上生成最终屏幕图像
# 4.1 图层创建的条件

Chorme 浏览器满足一下任意情况就会创建图层

  • 拥有 3D 变换的 CSS 属性
  • 使用加速视频解码的 video 节点
  • 使用 csnvas 节点
  • CSS3 动画节点
  • 拥有 CSS 加速属性的元素

# 5. 重绘

重绘是一个元素外观的改变所触发的浏览器行为，例如改变 outline，背景色等属性，浏览器会根据元素的新属性重新绘制，使元素呈现新的外观，重绘不会带来重新布局，所以并不一定伴随重排

  • 重绘和重排都是以图层为单位的，如果图层中某个元素需要重绘，那么整个图层都需要重绘
  • 我们一般把这些变化的东西拥有自己的一个图层
  • 大多数浏览器自己会为 CSS3 动画的节点自动创建图层

# 6. 重排

渲染对象在创建完成并且添加渲染树时，并不包含位置和大小信息，计算这些值的过程称为布局或者重排

  • 重绘不一定需要重排，改个颜色不会改变布局
  • 重排大多数情况下会导致重绘，改变一个网页元素的位置也需要重新绘制

# 5&6. 触发重绘重排的属性

触发重绘的属性（改变元素外观的属性）

- color - background - outline-color
- border-style - background-image - outline
- border-radius - background-position - outline-style
- visibility - background-repeat - outline-width
- text-decoration - background-size - box-shadow

触发回流的属性（改变元素布局的属性）

- width - top - text-align
- height - bottom - overflow-y
- padding - left - font-weight
- margin - right - overflow
- display - position - font-family
- border-width - float - line-height
- border - clear - vertival-align
- min-height

常见的触发重排的操作

重排的成本比重绘要高很多

一个节点的重排可能会导致子节点，父节点以及同级节点的重排

所以，一下动作成本是比较高的：

  • 当你增加，删除，修改 DOM 节点时，会导致重排和重绘
  • 当你移动 DOM 位置时
  • 当你 Resize 窗口的时候（移动端没有这个问题）
  • 当你修改网页的默认字体的时候
  • 获取某些属性的时候（width，height。。。）

优化方案：

  1. 元素位置移动变换时使用 CSS3 的 transform 来代替对 top，left 的操作，transform 和 opacity 的改变仅仅影响图层的组合（不会重排重绘）
  2. 使用 opacity 来代替 visibility
    • 使用 visibility 不触发重排，但是依然会触发重绘
    • 直接使用 opacity 即触发重绘又触发重排
    • opacity 配合图层使用，既不重绘又不重排（现在会重绘）
  3. 不要使用 table 布局 table-cell
  4. 将多次改变样式改变的操作合并成一次操作（不要一条一条的修改 DOM 的样式，预先定义号 class，然后修改 className）
  5. 将 DOM 离线后在修改（由于 display：none 的元素不在渲染树里面，对隐藏了的这个元素一顿操作也不会触发重绘重排）
  6. 利用文档碎片
  7. 不要把获取 DOM 节点的属性值放到循环里，不然循环多次就会重绘重排多少次
  8. 动画实现过程中，启用 GPU 硬件加速，transform： translateZ (0)
  9. 为动画元素新建图层，提供动画元素的 z-index
